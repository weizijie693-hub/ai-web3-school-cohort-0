# Web3 Network 基础

> 日期：2026-05-18
> 来源：AI×Web3 School Handbook — Network

## 核心理解

区块链网络的核心不是"存数据"，而是让互不信任的参与者对状态变化达成一致。

传统应用是直接写数据库，链上应用则是在一个公开、按区块推进、由网络共识维护的状态机上提交请求。交易需要经过：Wallet → RPC → Mempool → Block → Consensus → Execution → Confirmation → Indexing → Explorer 全链路。

## 关键概念

### Block（区块）
- 交易被批处理和排序的基本单位
- 包含：交易列表、前一个区块的引用、状态根、时间戳、Gas 用量、共识相关数据
- 交易顺序影响结果、Gas 限制决定吞吐量、每个新区块引用前一个形成可验证历史
- [Ethereum Blocks](https://ethereum.org/en/developers/docs/blocks/)

### Consensus（共识）
- 网络决定"哪个历史是有效的"的机制
- 不同节点收到的时间和内容不同，共识让它们对区块顺序和状态变更达成一致
- 影响：需要多少个确认才安全、可能发生短区块重组、RPC 异常时状态读取延迟

### PoS（权益证明）
- Ethereum 目前使用 PoS，验证者质押 ETH 来提议和证明区块
- 网络安全性来自经济质押、客户端实现和节点参与
- [Ethereum PoS](https://ethereum.org/en/developers/docs/consensus-mechanisms/pos/)

### Testnet（测试网）
- 模拟主网环境，测试合约、前端和交易流程
- 测试网资产无真实经济价值，用于验证：部署脚本、钱包连接、RPC 配置、合约调用、浏览器验证、前端状态处理
- **注意**：测试网不能替代主网安全审查（流动性、MEV、攻击动机、资产规模、用户行为都不同）

### L2（二层网络）
- 在主网之外执行大量交易，然后将结果或证明提交回主网
- 优势：更低费用、更快确认
- 复杂度：跨链桥、提现等待、排序器、跨链状态同步
- 产品设计不应只说"支持 Ethereum"，要明确当前网络、资产在哪个链、跨链桥时长

### Rollup
- 主流 L2 扩展方案，执行移到链下/L2，数据和结果提交到 L1
- 类型：Optimistic Rollup（乐观证明） 和 ZK Rollup（零知识证明）
- Rollup 降低了单笔交易成本，但没有消除链上系统复杂度

## AI × Web3 关联

AI Agent 读取链上状态或执行交易时，必须知道自己运行在哪个网络：
- 主网 vs 测试网，L1 vs L2，不同的 Chain ID 和合约地址
- 工具应返回结构化网络信息：Chain ID、RPC 来源、区块高度、交易哈希、确认数、浏览器链接
- Agent 的摘要应引用可验证数据，而非仅说"交易成功"

## 实操：测试网交易追踪

1. 获取测试网 ETH（水龙头）
2. 用测试钱包发送转账或调用合约
3. 在区块浏览器中查看：状态、区块号、Gas 用量、from/to 地址、日志
4. 记录从 pending 到 confirmed 的时间
5. 切换网络，观察余额和交易历史的差异
