# AI 基础概念共享词汇表

这份词汇表为团队在 Agent、Workflow、AI Coding 和 Tool Use 等方向上建立共同语言而编写，适合作为日常讨论和文档写作的参考基准。

## 大型语言模型 (LLM)

### 一句话解释
大型语言模型是一种基于海量文本数据训练而成的深度学习模型，能够根据输入的文本预测并生成连贯、上下文相关的自然语言输出。

### 具体例子
当你用 ChatGPT 写一封邮件，开头输入"尊敬的张总，关于上周我们讨论的合作方案"，模型会根据它从训练数据中学到的语言模式和知识，自动补写出得体、通顺的后续内容，而不需要你逐字逐句地写。

### 常见误解与使用边界
很多人误以为 LLM 是一个"知识库"或"搜索引擎"，实际上它并不存储事实，而是根据统计规律生成最可能的下文——这意味着它可能自信地编造出不存在的信息（即"幻觉"）。在做事实核查或引用具体数据时，必须验证模型的输出，不能直接采信。

## 提示 (Prompt)

### 一句话解释
提示是用户输入给 AI 模型的指令或问题，是人与模型进行交互和引导其输出的主要方式，决定了模型的行为方向和输出质量。

### 具体例子
在 Cursor 中写"用 React 写一个带搜索功能的用户列表组件，数据从 useUsers hook 获取"——这段自然语言描述就是一个 prompt，它告诉 AI 你想要什么功能、用什么技术栈、有什么约束条件。

### 常见误解与使用边界
有人认为 prompt 是"魔法咒语"，需要特定的神秘措辞才能生效。实际上，prompt 的本质是用清晰、具体的语言表达需求。好的 prompt 靠的是明确的上下文、约束条件和输出格式，而不是花哨的模板或所谓的"触发词"。

## 上下文窗口 (Context Window)

### 一句话解释
上下文窗口是模型在一次请求中能"看到"或"参考"的输入文本总量，决定了它在生成回复时可利用的信息上限。

### 具体例子
当你把整个项目的代码库粘贴到 Claude 的对话框中并询问"这个 bug 在哪"，如果粘贴的内容加上你的问题超过了上下文窗口限制，模型就会"忘记"最早输入的部分代码，导致回答遗漏关键信息。

### 常见误解与使用边界
许多人把上下文窗口当作"模型能记住的总信息量"，但它更像一个工作台面——所有信息只在当前对话中使用，对话结束后模型不会保留任何"记忆"。此外，填满上下文窗口不一定更好：过多的无关信息会稀释模型对关键内容的注意力，降低输出质量。

## 智能体 / Agent

### 一句话解释
智能体是一个能够自主感知环境、做出决策并采取行动来达成目标的 AI 系统，而不仅仅是被动地回答单个问题。

### 具体例子
一个代码审查 Agent 可以自动 checkout PR 分支、运行静态分析、调用 LLM 逐文件审查代码变更、在 PR 上留下逐条评论，并在发现严重问题时重新触发 CI——它全程自主决策，不需要人类逐一下达指令。

### 常见误解与使用边界
初学者常把"调 API 的脚本"和 Agent 混为一谈。Agent 的核心是具备自主推理和规划能力：它能在未知情况下降级处理、根据中间结果灵活调整策略。单纯的条件分支编排不属于 Agent，只是自动化脚本。Agent 的价值在于处理"你不知道会出什么状况"的任务。

## 工具使用 (Tool Use)

### 一句话解释
工具使用是指 AI 模型能够调用外部函数或 API（如执行代码、查询数据库、搜索网页）来获取信息或执行操作，从而突破自身纯文本生成能力的限制。

### 具体例子
当 AI 编程助手需要读取你的项目文件时，它并不是"凭空知道"文件内容——它通过调用 `read_file` 工具来获取。这就是 tool use：模型负责决定何时调用哪个工具以及传什么参数，工具负责执行并返回结果给模型做下一步推理。

### 常见误解与使用边界
有人误认为 tool use 是模型"学会"了操作数据库或发送网络请求。实际上，模型只是学会了决定"应该调用哪个工具和传什么参数"，工具的实际执行由外部运行时完成，模型自己并不具备这些能力。此外，模型不会持久化工具调用的结果，每次对话都需要重新获取。

## 护栏 (Guardrails)

### 一句话解释
护栏是在 AI 应用周围设置的安全边界和约束规则，确保模型的行为在预期范围内，防止产生有害、不当或不符合业务需求的输出。

### 具体例子
在一个客服 AI 中设置护栏：当用户询问"如何破解系统"时，模型不会尝试回答，而是回复"我无法提供这方面的信息，请咨询您的系统管理员"。这个行为不靠 prompt 里写 if-else，而是通过内容过滤、主题分类、输出验证等多层机制实现。

### 常见误解与使用边界
很多人以为在 system prompt 里加一句"你要做一个安全的助手"就是护栏了。实际上，有效的护栏需要在**输入层**（过滤恶意 prompt 注入）、**推理层**（约束输出内容的主题和风格）和**输出层**（正则匹配、PII 检测等后处理）叠加防御。单靠 prompt 指令很容易被越狱攻击绕过。

## 人工智能编码 (AI Coding)

### 一句话解释
AI Coding 是指开发者利用 AI 模型来辅助或自动完成代码编写、调试、重构和文档生成等工作，从根本上改变"人写机器读"的传统开发模式。

### 具体例子
在 VS Code 中使用 GitHub Copilot 或 Cursor：你在 TypeScript 文件中输入一个函数签名，AI 实时建议出完整的函数体；或者选中一段代码，输入"为这个函数添加单元测试，覆盖所有边界情况"——AI 自动生成测试文件，你只需审查和微调。

### 常见误解与使用边界
有人担心 AI Coding 会让开发者失业。实际上，当前的 AI Coding 是"副驾驶"而非"自动驾驶"——它能大幅提升写样板代码、写测试、改 bug 的效率，但在系统架构设计、业务合理性判断、安全审计和非功能性需求（性能、可维护性）的把控上，人的判断仍然不可替代。

## 追踪 (Tracing)

### 一句话解释
追踪是对 AI 应用一次请求从输入到输出的完整执行链路进行记录和监控，用于调试、性能分析和行为审计。

### 具体例子
当你的 AI Agent 执行一个复杂任务时，tracing 系统会记录完整的时间线：用户发了什么 prompt → 模型首次推理结果 → 调用了哪些工具（如搜索网页、读取文件）→ 每个工具的输入输出 → 模型根据工具结果做的下一步推理 → 最终输出。当 Agent 行为异常时，你可以沿着这条链路快速定位问题出在哪一步。

### 常见误解与使用边界
有人把 tracing 等同于普通的日志打印。但 tracing 强调的是**跨组件、跨调用的全链路关联**——它不仅记录"发生了什么"，还记录"发生在哪个请求上下文中"、"父子调用关系是什么"、"每步耗时多少"。对于单步问答场景，tracing 的价值不大；但在多步 Agent 调用中，它是问题排查的基础设施。

## 人在环中 (Human-in-the-Loop, HITL)

### 一句话解释
人在环中是指在 AI 系统的决策流程中保留人类介入环节，让人类对关键操作进行确认、审核或修正，而不是让 AI 全自动运行。

### 具体例子
一个 AI 邮件起草工具生成了一封回复客户的邮件，但它不会直接发送——而是将草稿展示在界面上，等你点击"确认发送"后才真正发出。这就是 HITL：AI 负责效率（快速起草），人类把控最终决策权（确认内容合适）。

### 常见误解与使用边界
HITL 不等于"每个微不足道的操作都要人点头"。如果 AI 每走一步都弹窗让人批准，那就失去了使用 AI 的意义。好的 HITL 设计需要识别哪些步骤**风险高**或**需要人类判断力**（如确认对外发送的内容、批准金融交易），哪些步骤可以**全自动**（如内部数据格式化）。

## 特工

### 一句话解释
特工是一个长期运行的自主 AI 程序，它在后台独立完成多步骤任务——无需人类实时介入，能自主规划、执行、纠错，最终交付可验证的成果。

### 具体例子
你给特工下达任务："每天上午 9 点抓取我关注的五个技术博客的新文章，用中文写 200 字以内的摘要发到我的 Slack 频道。"特工到点自动启动：抓取 RSS → 去重 → 调用 LLM 逐一总结 → 拼接成 Slack 消息发送。如果某个站点超时，它会自动重试并在日志中记录异常。

### 常见误解与使用边界
很多人以为"特工"和"智能体 / Agent"是同一个概念的两种翻译。它们的核心区别在于运行模式：**智能体**强调在一次交互中的自主推理能力（如对话中主动调用工具），而**特工**强调无人值守的长期运行（如后台的定时任务）。特工适合异步、批处理、定时触发的场景，不适用于需要实时对话或秒级响应的交互场景。
